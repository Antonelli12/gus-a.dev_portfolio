## ADDED Requirements

### Requirement: Portfolio leads with developer-turned-designer positioning
The portfolio SHALL communicate Gus as a developer turned designer with technical depth, product thinking, frontend implementation ability, and AI-aware workflow habits.

#### Scenario: Visitor reads first-page positioning
- **WHEN** a visitor reads the Home page hero and introductory copy
- **THEN** the primary identity is developer turned designer rather than generic frontend engineer or generic software engineer

#### Scenario: Visitor scans professional labels
- **WHEN** the site presents role-adjacent labels for Gus
- **THEN** labels may include Developer turned Designer, Product-first Engineer, Designer with technical depth, UX Engineer, Technical Product Designer, Product-minded Frontend Developer, and AI-compatible workflow practitioner

### Requirement: Engineering credibility supports the design direction
The portfolio SHALL keep software engineering and frontend implementation visible as proof of technical depth while making UX/product design the main direction of the narrative.

#### Scenario: Visitor reviews technical evidence
- **WHEN** a visitor reviews engineering skills or projects
- **THEN** the evidence supports implementation-aware design rather than replacing the developer-turned-designer positioning

### Requirement: Tone remains ambitious and credible
Portfolio wording SHALL be concise, public, credible, and ambitious without implying fake seniority, unsupported outcomes, or exaggerated expertise.

#### Scenario: Portfolio copy is written from CV-derived evidence
- **WHEN** CV-derived evidence is adapted into public portfolio copy
- **THEN** it avoids invented metrics, employers, seniority, case studies, achievements, and outcomes

### Requirement: AI-compatible workflow is framed as disciplined practice
The portfolio SHALL frame AI-compatible workflow as disciplined product and development practice involving workflow design, prompt iteration, AI-assisted debugging, edge-case exploration, Spec Driven Development, and human review.

#### Scenario: Visitor sees AI workflow messaging
- **WHEN** AI-related tools or workflows are mentioned
- **THEN** they are presented as part of structured delivery and judgement rather than as a replacement for design or engineering skill

#### Scenario: AI evidence is listed
- **WHEN** the portfolio lists AI-compatible workflow evidence
- **THEN** it may include Microsoft Copilot Studio, Azure AI Foundry, Azure services, Azure DevOps, Microsoft Copilot, Codex, agentic workflow design, prompt iteration, AI-assisted debugging, edge-case exploration, workflow design, and Spec Driven Development when supported by available evidence

### Requirement: CV evidence is adapted for portfolio presentation
The portfolio SHALL translate CV-derived evidence into portfolio-friendly page copy, project summaries, case-study framing, card descriptions, skill clusters, and proof points without pasting CV bullets directly.

#### Scenario: Source evidence becomes site content
- **WHEN** CV-derived experience is used to create portfolio content
- **THEN** the content distinguishes proven experience, current direction, and future ambition

### Requirement: Portfolio avoids generic repositioning traps
The portfolio MUST NOT become a generic UX portfolio with no technical differentiation, a generic frontend engineering portfolio, or an AI-buzzword portfolio disconnected from product and implementation evidence.

#### Scenario: Future content is reviewed
- **WHEN** new copy, cards, or case-study content is proposed
- **THEN** it is checked against the developer-turned-designer positioning and technical-depth differentiator
